Build a full-stack web app called “Constituent Touchpoint Planner.”

This should be a production-quality MVP for an internal NC State University Advancement administrative tool.

The primary goal of Constituent Touchpoint Planner is to help approved Alumni Engagement and Annual Giving staff upload donor ID-only campaign audiences, define planned communication touchpoints by channel and send date, check communication volume against user-defined thresholds, apply campaign-specific suppressions and seed IDs, generate final donor ID-only send lists, and store exported touchpoints for future communication volume reference.

This app is not a CRM, not a marketing automation system, and not a PII storage system. It is a donor ID-only campaign communication volume planning and touchpoint tracking tool.

The app should help answer:
- Which Donor IDs are scheduled to receive communications?
- How many touchpoints will each Donor ID receive in a selected timeframe?
- How many touchpoints will each Donor ID receive by channel?
- How many touchpoints will each Donor ID receive by campaign type?
- Which Donor IDs exceed user-defined communication thresholds?
- Which Donor IDs should be suppressed from specific sends, channels, or campaign types?
- Which final Donor IDs should be exported for each send?
- What communication touchpoints have already been exported/sent and should count toward future checks?
- What is the communication volume by Donor ID, channel, campaign type, date, campaign, and submitting user?

Optimize the application around accurate Donor ID normalization, clean no-PII workflows, reliable threshold checking, clear preview counts, accessible review/export workflows, and durable touchpoint history.

Core workflow:
Login → Create campaign → Upload donor ID audience via CSV or Google Sheet URL → Map/confirm donor ID column → Validate/normalize/deduplicate IDs → Build planned touchpoints → Create threshold checks → Preview projected volume → Review flagged IDs → Apply campaign-specific suppressions and seed IDs → Generate one final donor ID-only CSV per send → Mark exported touchpoints as sent/planned in the database → Use exported touchpoints in future threshold checks.

NO-PII DATA HANDLING
────────────────────────────────────────

This app must be designed around a strict no-unnecessary-PII principle.

The app should use Donor ID as the primary and only constituent identifier.

Do not ask for, store, display, or export:
- names
- email addresses
- phone numbers
- mailing addresses
- giving amounts
- employment information
- personal notes about constituents
- any other unnecessary personal information

Allowed constituent-level field:
- normalized Donor ID only

Donor ID normalization:
- Donor IDs from Lobo/Salesforce are numeric.
- Donor IDs may be uploaded with or without leading zeroes.
- Normalize all Donor IDs to exactly 8 digits by left-padding with zeroes.
- Examples:
  - 258155 → 00258155
  - 119454 → 00119454
  - 1017955 → 01017955
  - 00258155 → 00258155
- Reject IDs longer than 8 digits.
- Reject IDs containing letters or special characters.
- Trim leading/trailing whitespace before validation.
- Blank rows should be ignored and counted as skipped rows.
- Preserve leading zeroes in all displays and exports.

Upload requirements:
- Main audience uploads should contain one Donor ID column.
- CSV uploads may have a header row or no header row.
- Users should be able to map/confirm the Donor ID column before processing.
- Accept likely column names such as:
  - donor_id
  - Donor ID
  - LOBO ID
  - Lobo ID
  - Salesforce ID
  - Constituent ID
  - ID
- If extra columns are present, ignore them and do not store them.
- Show a warning:
  “Only Donor ID values were processed. Extra columns were ignored and not stored.”
- Source upload files should be deleted after processing.
- Store only normalized Donor IDs and campaign/touchpoint metadata.

PII detection:
Add basic PII warning support for campaign notes, internal notes, and suppression notes.

When a user enters notes, check for common PII patterns such as:
- email address pattern
- phone number pattern
- street address-like pattern

If detected, show a non-blocking warning:
“Please avoid entering names, phone numbers, email addresses, or other unnecessary PII. This system should use Donor ID only.”

Allow the user to revise before saving.

First-login or recurring acknowledgement:
Add a simple acknowledgement modal or banner reminding users:
“Use Donor ID only. Do not upload or enter unnecessary PII.”

No-PII helper text:
Add visible reminders on:
- upload screens
- notes fields
- suppression screens
- export screens

Suggested helper text:
“Use Donor ID only. Do not upload or enter names, phone numbers, email addresses, addresses, giving amounts, or other unnecessary PII.”

CORE DATA MODEL CONCEPT
────────────────────────────────────────

A touchpoint equals:
One planned send to one Donor ID.

Example:
If 10,000 Donor IDs receive 3 emails, that creates 30,000 touchpoint records after export.

For the MVP:
- One campaign has one primary uploaded audience.
- Each campaign can have multiple planned sends/touchpoints.
- Each touch applies to the full uploaded audience by default.
- IDs can later be removed by channel/send-specific suppression.
- Do not support multiple audience subsets per touch in the first version.
- This may be revisited later.

THRESHOLD / COMMUNICATION VOLUME CHECKER
────────────────────────────────────────

Users create thresholds each time during campaign setup.

Admin panel can support reusable/global threshold templates later, but global default thresholds should not be turned on by default.

Threshold types:
Users should be able to test:
- Max total touchpoints in X days
- Max touchpoints by channel in X days
- Max touchpoints by campaign type in X days
- Max touchpoints by channel + campaign type in X days
- Max solicitations in X days
- Max emails in X days
- Max texts in X days
- Max mail touches in X days

Threshold filters:
Users should be able to include:
- All campaign types
- Specific campaign types
- All channels
- Specific channels
- Channel + campaign type combinations

Threshold mode:
Use any rolling window including the proposed touch.

The UI must clearly explain this. Include text like:
“A 14-day threshold checks any 14-day period that includes each selected send date. This includes other planned touches in this campaign and previously exported touchpoints.”

Important threshold rules:
- The proposed campaign’s own planned touches should count against itself.
- Exported / Sent touchpoints from previous campaigns count.
- Archived campaigns still count.
- Voided campaigns do not count.
- Deleted campaigns do not count.
- Draft, Uploaded, Previewed, and Finalized campaigns do not count until exported.
- Seed IDs do not count toward thresholds.
- Suppressed donor touches should not count in final post-suppression export counts.
- Because suppressions are applied near the end, the app should show both pre-suppression projected counts and post-suppression final export counts.

Default threshold action:
- Flag only.
- Do not remove flagged IDs unless the user chooses to suppress/remove them.

User actions for threshold results:
- Track only, do not suppress
- Flag only
- Automatically remove flagged IDs
- Manually review flagged IDs
- Override flagged IDs

Standard users can override threshold flags.

No override reason is required.

Threshold preview should include clear explanations:
Example:
“Donor ID 00258155 is projected to receive 6 email touchpoints in a 14-day rolling window. Your selected threshold is 5 email touchpoints in 14 days.”

Live recalculation:
- Counts should update live as users adjust threshold settings.
- If full live recalculation is expensive, use a clear “Recalculate Preview” button and accessible loading state.

Suppression behavior:

- Suppressed touchpoints should not be stored as sent/included touchpoint records.

- Suppression reason is optional.
- Suppression notes should include no-PII warning and basic PII pattern detection.

Seed ID input options:
- CSV upload
- Google Sheet URL
- Manual paste box

Seed behavior:
- Seed IDs are Donor IDs.
- Seed IDs must pass the same 8-digit validation/normalization.
- Invalid seed IDs should be rejected and downloadable.
- Seed IDs can be added to:
  - all exports
  - specific channel exports
  - specific send/touch exports
- Seed IDs appear at the bottom of exported CSVs.
- Seed IDs should not count toward thresholds.
- Seed IDs should not create stored donor touchpoint records.
- Seed IDs should not be stored as constituent communication history.
- Seed IDs do not need to be stored in export history for MVP.
- Deduplicate seed IDs within each export.
- If a seed ID already exists in the audience, do not duplicate it in the export.

Each final send-list CSV should contain Donor IDs only.

- Include only donor_id column.
- Preserve leading zeroes.
- No names.

Administrative/reporting exports may include:

- +
- -
- @
- tab
- carriage return

- final channel/send exports
- rejected ID exports

- suppressed ID exports

- audit exports

THRESHOLD CALCULATION DETAILS
────────────────────────────────────────

Threshold calculation must consider:
1. Existing exported/sent touchpoint records that count toward thresholds.
2. Proposed touches in the current campaign.
3. Selected threshold filters by channel and/or campaign type.
4. Rolling windows that include each proposed touch.

Use “any rolling window including the proposed touch” logic.

Example:
If the user creates a threshold:
“Max 5 email touchpoints in 14 days”

For each Donor ID and each proposed email touch:
- Find all email touchpoints for that Donor ID that count toward thresholds.
- Include existing exported/sent records.
- Include proposed touches in the current campaign.
- Evaluate any 14-day window that includes the proposed touch.
- If count > 5, flag the Donor ID and explain why.

The explanation should be human-readable:
“Donor ID 00258155 is projected to receive 6 email touchpoints between September 1, 2026 and September 14, 2026. The selected threshold is 5 email touchpoints in 14 days.”

Make sure users understand rolling window logic in the UI.
